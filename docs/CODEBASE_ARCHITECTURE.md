# Oykos Triage AI — Codebase & Architecture Reference

A precise, step-by-step description of how the system works: what goes in, what happens, and what comes out at each stage.

---

## Overview

A parent sends a WhatsApp-style message in Italian describing their child's symptoms. The system reads it and returns a structured list of matched symptoms from a fixed catalog of 80 canonical labels.

There are two modes:

- **Baseline** (default, production): one LLM call with the full catalog in the system prompt
- **Pipeline**: five-stage retrieval + extraction flow for cases the baseline can't cover

Both modes produce the same output format.

---

## The Symptom Catalog

**File:** `data/catalog/symptom_catalog.json`

80 symptoms, each with:

| Field | Example |
|---|---|
| `code` | `SI001` |
| `label_it` | `Febbre` |
| `label_en` | `Fever` |
| `triage_depth` | `Alta` or `Media` |
| `short_definition` | Clinical definition in Italian |

If `symptom_extraction/catalog/enriched_catalog.json` exists, it is used instead. Enriched entries add `examples_it`, `synonyms_it`, `negation_patterns_it`, and `disambiguation` — generated by `scripts/enrich_catalog.py`.

The catalog is loaded once at startup by `symptom_extraction/catalog/loader.py` and passed as a Python list through the entire system. Nothing reads the JSON file at request time.

---

## Baseline Mode (Default)

**Entry point:** `PipelineOrchestrator.run_baseline()` in `symptom_extraction/pipeline/orchestrator.py`

### Step 1 — Build the system prompt

**Code:** `symptom_extraction/prompts/templates.py → build_baseline_prompt()`

**Context going in:** The full 80-symptom catalog

**What it does:** Formats every symptom as a labeled block (code, Italian label, English label, triage depth, definition, examples, synonyms, negation patterns, disambiguation) and embeds them all into a fixed system prompt template.

**The prompt contains:**
- Role: "You are a pediatric symptom extractor"
- 10 extraction rules (negation exclusion, past-resolved exclusion, hedged inclusion, evidence span requirement, completeness requirement, etc.)
- Italian negation markers list
- Italian temporal resolution markers list
- The full 80-symptom catalog block

**Approximate size:** ~27,000 characters (~9,500 tokens)

**Output:** A single string — the complete system prompt. Built once at startup, reused for every request.

---

### Step 2 — Build the constrained output schema

**Code:** `symptom_extraction/models.py → build_extraction_model()`

**Context going in:** List of all 80 `(code, label_it)` pairs

**What it does:** Dynamically builds a Pydantic model where `code` is a `Literal` type constrained to exactly the 80 valid codes, and `label_it` is a `Literal` type constrained to the 80 valid Italian labels. This schema is serialized to JSON Schema and sent to the LLM API.

**Why this matters:** The model cannot output a code or label that isn't in the enum — hallucinated symptoms are mechanically impossible at the API level.

**Output schema shape:**
```json
{
  "symptoms": [
    {
      "code": "<one of SI001–SI080>",
      "label_it": "<matching Italian label>",
      "evidence_span": "<verbatim substring from parent message>",
      "negated": false,
      "hedged": false,
      "temporal_status": "current | past_resolved | chronic",
      "confidence": "high | medium | low",
      "onset": "<temporal phrase or null>"
    }
  ],
  "excluded": [
    {
      "code": "<SI001–SI080>",
      "label_it": "<Italian label>",
      "reason": "negated | past_resolved | below_threshold",
      "evidence_span": "<verbatim substring or null>"
    }
  ]
}
```

---

### Step 3 — LLM call

**Code:** `symptom_extraction/llm/openai_provider.py` or `anthropic_provider.py`

**Context going in:**
- System prompt (from Step 1) — static, cached after first call
- User message — the parent's Italian message

**Method:**
- **OpenAI:** `response_format` with strict JSON schema mode. The schema is the constrained Pydantic model from Step 2.
- **Anthropic:** `tool_use` with `tool_choice` forced to a single tool. The tool's `input_schema` is the constrained model from Step 2. System prompt has `cache_control: {"type": "ephemeral"}` for prompt caching.

**Parameters:**
- `temperature: 0.0` (deterministic; skipped for Opus models that don't accept it)
- `max_tokens: 4096`
- `model`: set in `config.yaml` (e.g. `gpt-5.5-2026-04-23`, `claude-sonnet-4-6`)

**Retry logic:** 3 attempts, exponential backoff (2s → 4s → 8s) via `tenacity`

**Expected output:** A validated Pydantic object matching the constrained schema above

---

### Step 4 — Post-processing

**Code:** `orchestrator.py → run_baseline()` (lines 86–112)

**Context going in:** The parsed Pydantic object from the LLM

**What it does:** Moves any symptoms marked `negated=true` or `temporal_status=past_resolved` from `symptoms` into `excluded`. The LLM is instructed to handle this itself, but post-processing enforces it regardless of what the model returned.

**Output:** `GatedResult` — the final structured response

---

### Final Output (`GatedResult`)

```python
GatedResult(
    confirmed=[ExtractedSymptom(...)],       # symptoms to act on
    flagged_for_review=[],                   # always empty in baseline
    abstained=[],                            # always empty in baseline
    excluded=[ExcludedSymptom(...)],         # negated / past-resolved
    pipeline_metadata=PipelineMetadata(
        mode="baseline",
        latency_ms=1200.0,
        llm_calls=1,
        total_input_tokens=9700,
        total_output_tokens=180,
    )
)
```

**Tokens per call:**
- First call: ~9,500 input (catalog) + 50–200 (message) + 100–400 output
- Subsequent calls: 50–200 input (cache hit on catalog) + 100–400 output

## Pipeline Mode

Used when baseline accuracy on specific symptoms is insufficient. Five stages; only Stages 2c and 4 make LLM calls.

**Entry point:** `PipelineOrchestrator.run_pipeline()`

---

### Stage 1 — Clause Segmentation

**Code:** `symptom_extraction/pipeline/stage1_segmentation.py → RuleBasedSegmenter`

**Context going in:** The raw parent message

**Method:** Rule-based. Splits on Italian conjunctions (`, e`, `, ma`, `, però`, etc.), sentence boundaries (`. `), and semicolons using a compiled regex. Short messages (<60 chars with no punctuation) are returned as a single clause. Each part is annotated with whether it contains a negation marker and any temporal marker found.

**Why clauses and not keywords:** Negation travels with its clause. "Non ha la febbre" is a negated fever signal as a whole — stripping to keywords destroys that. "La febbre di ieri è passata" is a resolved symptom — only the full clause makes that clear.

**Output:** List of `Clause(text, has_negation, temporal_marker)`

Example:
```
In:  "Da ieri sera ha la febbre alta, non vuole mangiare, e credo gli faccia male la pancia."

Out: [
  Clause(text="Da ieri sera ha la febbre alta", temporal_marker="da ieri"),
  Clause(text="non vuole mangiare", has_negation=True),
  Clause(text="credo gli faccia male la pancia")
]
```

---

### Stage 2 — Parallel Retrieval (per clause, 3 methods)

Each clause is queried against all three retrievers. Dense and LLM retrieval run concurrently (`asyncio.gather`); BM25 runs synchronously (pure compute, no I/O).

#### 2a — Dense Retrieval

**Code:** `symptom_extraction/pipeline/stage2a_dense.py`

**Context going in:** Clause text, pre-computed symptom embedding vectors

**Method:** Embeds the clause with `intfloat/multilingual-e5-large-instruct` (SentenceTransformers). Computes cosine similarity against pre-computed vectors for all 80 symptoms. Each symptom is embedded as a concatenation of its label, definition, examples, and synonyms. The index is built once with `python cli.py build-index`.

**Output:** Top-10 `(code, cosine_score)` pairs ranked by similarity

**Catches:** Paraphrases and semantic matches — e.g. "scotta tantissimo" → Febbre even without the word "febbre"

---

#### 2b — BM25 Retrieval

**Code:** `symptom_extraction/pipeline/stage2b_bm25.py`

**Context going in:** Clause text, synonym lexicon built from catalog

**Method:** BM25 over a lexicon of surface forms per symptom (label + synonyms + examples). Italian lemmatization via `simplemma`. No LLM involved.

**Output:** Top-10 `(code, bm25_score)` pairs

**Catches:** Exact and morphological surface matches that dense retrieval misses — e.g. "ha la febbre" matching exactly on the token "febbre"

---

#### 2c — LLM Extract + Nearest Neighbor

**Code:** `symptom_extraction/pipeline/stage2c_llm_extract.py`

**Context going in:** Clause text

**Prompt:**
```
List every symptom, clinical sign, or physical complaint mentioned in this
Italian parent message. Include explicitly stated symptoms, implied symptoms,
and negated symptoms. Return a JSON array of {"phrase": "...", "negated": true/false}.
```

**Method:** Small LLM (cheap model, e.g. `gpt-4o-mini`) extracts free-form symptom phrases from the clause. Each phrase is then embedded and nearest-neighbor matched against the symptom catalog vectors (reusing the dense index).

**Output:** Top-5 `(code, similarity_score)` pairs per extracted phrase, merged

**Catches:** Implied symptoms with no surface match — e.g. "non riesce ad alzarsi dal letto" → Debolezza/Astenia even though neither word appears

---

### Stage 3 — RRF Fusion

**Code:** `symptom_extraction/pipeline/stage3_fusion.py → RRFFusion`

**Context going in:** Three ranked lists per clause (dense, BM25, LLM), across all clauses

**Method:** Reciprocal Rank Fusion — pure Python, no LLM. For each candidate symptom `s`:

```
score(s) = Σ over all (clause, retriever) pairs:  1 / (60 + rank_of_s_in_that_list)
```

Candidates that ranked high in multiple retrievers and/or across multiple clauses get the highest scores. The `k=60` constant dampens rank differences at the top.

**Output:** Top-15 `CandidateSymptom` objects ranked by RRF score, each annotated with which retrievers and clauses surfaced it

**Why not just union:** A label ranked #1 in two retrievers should outrank one ranked #15 once. Naive union throws that signal away.

---

### Stage 4 — Constrained Extraction

**Code:** `symptom_extraction/pipeline/stage4_extraction.py`

**Context going in:**
- The original unmodified parent message (full context for negation/temporality judgment)
- The top-15 candidate symptoms with their full catalog entries (definition, examples, synonyms, negation patterns)
- A constrained output schema limited to just those 15 candidates

**Prompt:** Same structure as baseline (`STAGE4_SYSTEM_PROMPT`) but the catalog block contains only the 15 candidates, not all 80. The enum constraint is also narrowed to those 15 codes.

**Method:** Same as baseline Step 3 — frontier LLM with forced structured output

**Why the original message:** Segmentation was for retrieval only. The final extraction decision (is this negated? is this past-resolved? is this actually asserted?) needs the full message context.

**Why only 15 candidates:** Smaller enum = less surface area for false positives. The model can only emit labels that retrieval already surfaced as plausible.

**Output:** `ExtractionResult(symptoms=[...], excluded=[...])`

---

### Stage 5 — Confidence Gating

**Code:** `symptom_extraction/pipeline/stage5_gating.py → ConfidenceGate`

**Context going in:** Extracted symptoms (with LLM confidence levels) + retrieval metadata (which retrievers surfaced each symptom, RRF score)

**Method:** Rule-based score per symptom:

```
base_score = 1.0 (high) | 0.6 (medium) | 0.3 (low)
+ 0.15 per additional retriever that agreed (max +0.30)
+ 0.10 if RRF score > 0
capped at 1.0
```

**Routing:**
- `score ≥ 0.7` → **confirmed** (send to triage)
- `0.4 ≤ score < 0.7` → **flagged_for_review** (human clinician reviews)
- `score < 0.4` → **abstained** (dropped)

**Output:** `GatedResult` with symptoms split into confirmed / flagged / abstained / excluded

## Data Flow Summary

```
Parent message (Italian string)
        │
        ├─── BASELINE MODE ───────────────────────────────────────────────┐
        │                                                                   │
        │   System prompt: instructions + all 80 symptoms (~27k chars)     │
        │   User message:  the parent's text                               │
        │   Schema:        Literal enum constrained to 80 codes            │
        │   LLM call:      1 (frontier model, temperature=0)               │
        │   Post-process:  move negated/past-resolved to excluded          │
        │                                                                   │
        └─── PIPELINE MODE ───────────────────────────────────────────────┤
                                                                            │
            Stage 1  Rule-based clause split (no LLM)                      │
            Stage 2  Per clause:                                            │
                       2a Dense embedding similarity (no LLM)              │
                       2b BM25 lexical match (no LLM)                      │
                       2c Small LLM → free phrases → embedding NN          │
            Stage 3  RRF fusion → top-15 candidates (no LLM)              │
            Stage 4  Frontier LLM with 15-item constrained schema          │
            Stage 5  Rule-based confidence gate (no LLM)                   │
                                                                            ▼
                                                               GatedResult
                                                               ├── confirmed:        [ExtractedSymptom]
                                                               ├── flagged_for_review:[ExtractedSymptom]
                                                               ├── abstained:        [ExtractedSymptom]
                                                               ├── excluded:         [ExcludedSymptom]
                                                               └── pipeline_metadata: latency, token counts
```

## Configuration

`config.yaml` controls all runtime behaviour. Override any field with a CLI flag or environment variable.

```yaml
provider: openai              # openai | anthropic | gemini
mode: baseline                # baseline | pipeline

openai:
  frontier_model: gpt-5.5-2026-04-23
  cheap_model: gpt-4o-mini

anthropic:
  frontier_model: claude-sonnet-4-6
  cheap_model: claude-3-5-haiku-20241022

pipeline:
  top_k_candidates: 15        # candidates passed to Stage 4
  rrf_k_constant: 60          # RRF dampening constant

gating:
  high_threshold: 0.7         # confirmed if score >= this
  review_threshold: 0.4       # flagged if score >= this, else abstained

temperature: 0.0
cache_system_prompt: true
use_verifier: false           # optional second LLM pass (disabled — hurts recall)
max_concurrency: 5            # parallel evaluation batch size
```

## Invariants Worth Knowing

**Hallucinated labels are impossible.** The `code` and `label_it` fields are `Literal` types in the JSON schema. The model cannot emit a token outside the allowed set — this is enforced at the API layer.

**Temperature is omitted for Opus models.** `claude-opus-4-5`, `claude-opus-4-6`, `claude-opus-4-7`, `claude-opus-4-8` reject the `temperature` parameter. The provider detects this from `_NO_TEMPERATURE_MODELS` and silently omits it.

**The verifier is disabled.** `use_verifier: false` in config. A second LLM pass to strip consequence symptoms (e.g. insomnia caused by toothache) caused more harm than good — it increased false negatives faster than it reduced false positives.

**Catalog is the bottleneck, not the prompt.** The accuracy ceiling for any symptom is set by whether the catalog has enough signal for the model to recognize it. Prompt changes help at the margins; catalog enrichment (more examples, better disambiguation) is where accuracy gains come from.
